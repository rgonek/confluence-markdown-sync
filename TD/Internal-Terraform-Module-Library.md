# Internal Terraform Module Library

Luminary's infrastructure is provisioned via a set of internal Terraform modules maintained in the [luminary/infra](https://github.com/luminary/infra) repository under `modules/`. These modules wrap AWS resources with Luminary-specific defaults, enforce tagging standards, and expose a consistent interface that reduces boilerplate across environments.

This page documents each module: its purpose, inputs, outputs, usage example, and any known caveats. For cluster-level context on how these modules are composed, see the [Kubernetes Cluster Setup](Production-Kubernetes-Cluster-Setup-(EKS).md) page.

---

## Module Versioning

Modules are versioned using Git tags in the `luminary/infra` repository. Tags follow the format `modules/<module-name>/vMAJOR.MINOR.PATCH`. All consuming Terraform stacks (in `stacks/production/`, `stacks/staging/`, etc.) must reference a specific tag — floating references to `main` are prohibited in production stacks.

```hcl
module "vpc" {
  source  = "git::https://github.com/luminary/infra.git//modules/luminary-vpc?ref=modules/luminary-vpc/v2.3.1"
  # ...
}
```

Renovate Bot runs in the `luminary/infra` repository and opens PRs to bump module references when new versions are tagged. PRs are auto-approved if CI passes and the bump is patch-level; minor and major bumps require a Platform Engineering review.

## Contributing New Modules

Before writing a new module:

1. Check whether an existing module can be extended with a new input variable.
2. If a new module is warranted, create a directory under `modules/<module-name>/` with the standard structure:

   ```
   modules/luminary-newmodule/
   ├── main.tf
   ├── variables.tf
   ├── outputs.tf
   ├── versions.tf
   └── README.md    # auto-generated by terraform-docs
   ```
3. Use `terraform-docs` to generate the `README.md` (`make docs` in the module directory).
4. Add at least one test in `modules/luminary-newmodule/tests/` using Terratest.
5. Tag the initial release as `modules/luminary-newmodule/v1.0.0`.

All modules must comply with the [Module Standards](https://wiki.luminaryapp.com/infra/terraform-module-standards) (internal link), which cover tagging requirements, provider version pinning, and sensitive output handling.

## `luminary-vpc`

Provisions the standard Luminary VPC with public, private (EKS), and data subnet tiers across all three AZs. Attaches an Internet Gateway, NAT Gateways (one per AZ), and a set of standard VPC Flow Log configurations.

**Source:** `modules/luminary-vpc` | **Latest:** `v2.3.1`

### Inputs

| Variable | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| `environment` | `string` | — | Yes | Environment name (`production`, `staging`, `dev`). Used in resource names and tags. |
| `vpc_cidr` | `string` | `"10.0.0.0/16"` | No | VPC CIDR block. |
| `public_subnet_cidrs` | `list(string)` | — | Yes | List of 3 CIDRs for public subnets (one per AZ). |
| `private_eks_subnet_cidrs` | `list(string)` | — | Yes | List of 3 CIDRs for private EKS subnets. |
| `data_subnet_cidrs` | `list(string)` | — | Yes | List of 3 CIDRs for data-tier subnets. |
| `enable_flow_logs` | `bool` | `true` | No | Create VPC Flow Logs to CloudWatch. |
| `flow_log_retention_days` | `number` | `30` | No | CloudWatch log group retention. |
| `single_nat_gateway` | `bool` | `false` | No | Use a single NAT Gateway (cost saving for non-production environments). |

### Outputs

| Output | Type | Description |
| --- | --- | --- |
| `vpc_id` | `string` | ID of the created VPC. |
| `public_subnet_ids` | `list(string)` | IDs of public subnets. |
| `private_eks_subnet_ids` | `list(string)` | IDs of private EKS subnets. |
| `data_subnet_ids` | `list(string)` | IDs of data-tier subnets. |
| `nat_gateway_ips` | `list(string)` | Elastic IPs of NAT Gateways (for allowlisting). |

### Usage

```hcl
module "vpc" {
  source = "git::https://github.com/luminary/infra.git//modules/luminary-vpc?ref=modules/luminary-vpc/v2.3.1"

  environment = "production"
  vpc_cidr    = "10.0.0.0/16"

  public_subnet_cidrs = [
    "10.0.0.0/24",
    "10.0.2.0/24",
    "10.0.4.0/24",
  ]

  private_eks_subnet_cidrs = [
    "10.0.10.0/22",
    "10.0.14.0/22",
    "10.0.18.0/22",
  ]

  data_subnet_cidrs = [
    "10.0.30.0/24",
    "10.0.31.0/24",
    "10.0.32.0/24",
  ]

  enable_flow_logs         = true
  flow_log_retention_days  = 30
  single_nat_gateway       = false
}
```

### Known Caveats

- **NAT Gateway cost:** Three NAT Gateways cost ~$135/month (plus data transfer). `single_nat_gateway = true` is available for staging to reduce cost, but **must not** be used in production — a NAT Gateway failure in one AZ would take down all outbound traffic from that AZ's EKS nodes.
- **Subnet CIDR sizing:** The private EKS subnets use /22 (1,022 usable IPs each). With `ENABLE_PREFIX_DELEGATION` on the VPC CNI, each EKS node consumes a /28 prefix (16 IPs) per assigned prefix. Do not shrink these below /22 without recalculating pod density requirements.
- **IPv6:** The module does not currently support IPv6. There is an open issue to add dual-stack support (tracked in `PLAT-841`).

## `luminary-eks`

Provisions an EKS cluster with managed node groups, OIDC provider for IRSA, and all standard cluster add-ons (VPC CNI, CoreDNS, kube-proxy, EBS CSI driver). Does not install application-level add-ons (Istio, cert-manager, etc.) — those are managed via ArgoCD.

**Source:** `modules/luminary-eks` | **Latest:** `v3.1.0`

### Inputs

| Variable | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| `cluster_name` | `string` | — | Yes | EKS cluster name. |
| `kubernetes_version` | `string` | — | Yes | Target Kubernetes version (e.g., `"1.30"`). |
| `vpc_id` | `string` | — | Yes | VPC ID (output of `luminary-vpc`). |
| `subnet_ids` | `list(string)` | — | Yes | Private EKS subnet IDs for worker nodes. |
| `control_plane_subnet_ids` | `list(string)` | — | Yes | Subnets for control plane ENIs (usually same as worker subnets). |
| `node_groups` | `map(object)` | — | Yes | Map of node group configurations (see schema below). |
| `cluster_admin_role_arns` | `list(string)` | `[]` | No | IAM role ARNs granted `system:masters` in `aws-auth`. |
| `enable_karpenter` | `bool` | `false` | No | Create Karpenter IAM role and EC2 NodeClass prerequisites. |
| `log_types` | `list(string)` | `["api","audit","authenticator"]` | No | EKS control plane log types to enable. |

`node_groups` **object schema:**

```hcl
node_groups = {
  "<name>" = {
    instance_types  = list(string)   # e.g., ["m6i.2xlarge"]
    min_size        = number
    max_size        = number
    desired_size    = number
    disk_size_gb    = number
    labels          = map(string)
    taints          = list(object({ key, value, effect }))
    ami_type        = string         # AL2_x86_64 | AL2_ARM_64 | etc.
    capacity_type   = string         # ON_DEMAND | SPOT
  }
}
```

### Outputs

| Output | Type | Description |
| --- | --- | --- |
| `cluster_name` | `string` | EKS cluster name. |
| `cluster_endpoint` | `string` | API server endpoint URL. |
| `cluster_ca_certificate` | `string` | Base64-encoded CA cert for kubeconfig. |
| `oidc_provider_arn` | `string` | OIDC provider ARN for IRSA. |
| `oidc_provider_url` | `string` | OIDC provider URL (without `https://`). |
| `node_role_arn` | `string` | IAM role ARN for worker nodes. |
| `karpenter_role_arn` | `string` | Karpenter IAM role ARN (if `enable_karpenter = true`). |

### Usage

```hcl
module "eks" {
  source = "git::https://github.com/luminary/infra.git//modules/luminary-eks?ref=modules/luminary-eks/v3.1.0"

  cluster_name        = "luminary-production"
  kubernetes_version  = "1.30"
  vpc_id              = module.vpc.vpc_id
  subnet_ids          = module.vpc.private_eks_subnet_ids
  control_plane_subnet_ids = module.vpc.private_eks_subnet_ids

  cluster_admin_role_arns = [
    "arn:aws:iam::123456789012:role/luminary-platform-eng-admin",
  ]

  enable_karpenter = true

  node_groups = {
    system = {
      instance_types = ["m6i.large"]
      min_size       = 3
      max_size       = 6
      desired_size   = 3
      disk_size_gb   = 30
      capacity_type  = "ON_DEMAND"
      labels         = { "luminary.io/node-pool" = "system" }
      taints         = []
      ami_type       = "AL2_x86_64"
    }
    app = {
      instance_types = ["m6i.2xlarge"]
      min_size       = 6
      max_size       = 40
      desired_size   = 8
      disk_size_gb   = 50
      capacity_type  = "ON_DEMAND"
      labels         = { "luminary.io/node-pool" = "app" }
      taints         = []
      ami_type       = "AL2_x86_64"
    }
  }
}
```

### Known Caveats

- `aws-auth` **ConfigMap management:** The module manages the `aws-auth` ConfigMap to map the node IAM role to `system:nodes`. If you manually edit `aws-auth` outside of Terraform, the next `terraform apply` will overwrite your changes. Use the `cluster_admin_role_arns` input instead.
- **Add-on version pinning:** The module pins managed add-on versions (`vpc-cni`, `coredns`, `kube-proxy`, `aws-ebs-csi-driver`) to the latest version compatible with the specified `kubernetes_version`. Bumping `kubernetes_version` will also bump add-on versions on the next apply. Always check the EKS add-on changelog before applying a Kubernetes version bump.
- **Karpenter prerequisites:** Setting `enable_karpenter = true` creates the IAM role and the EC2 `NodeClass` prerequisites, but does not install the Karpenter Helm chart. That is managed by ArgoCD in `argocd/apps/karpenter.yaml`.

## `luminary-rds`

Provisions an RDS PostgreSQL instance (or Multi-AZ cluster) with standard Luminary security group rules, parameter groups, and enhanced monitoring.

**Source:** `modules/luminary-rds` | **Latest:** `v1.4.2`

### Inputs

| Variable | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| `identifier` | `string` | — | Yes | RDS instance identifier (used in endpoint hostname). |
| `environment` | `string` | — | Yes | Environment name. |
| `engine_version` | `string` | `"15.6"` | No | PostgreSQL engine version. |
| `instance_class` | `string` | — | Yes | RDS instance class (e.g., `"db.r6g.2xlarge"`). |
| `multi_az` | `bool` | `true` | No | Enable Multi-AZ standby. |
| `subnet_ids` | `list(string)` | — | Yes | Subnet IDs for the DB subnet group (data subnets). |
| `vpc_id` | `string` | — | Yes | VPC ID for security group. |
| `allowed_security_group_ids` | `list(string)` | `[]` | No | Security group IDs allowed to connect on port 5432. |
| `allocated_storage_gb` | `number` | `100` | No | Initial storage allocation in GB. |
| `max_allocated_storage_gb` | `number` | `1000` | No | Maximum autoscaling storage limit. |
| `backup_retention_days` | `number` | `14` | No | Automated backup retention period. |
| `deletion_protection` | `bool` | `true` | No | Protect against accidental deletion. |
| `parameter_group_family` | `string` | `"postgres15"` | No | RDS parameter group family. |
| `extra_parameters` | `map(string)` | `{}` | No | Additional PostgreSQL parameters to set. |

### Outputs

| Output | Type | Description |
| --- | --- | --- |
| `endpoint` | `string` | RDS endpoint hostname (writer). |
| `port` | `number` | Database port (5432). |
| `db_security_group_id` | `string` | ID of the RDS security group. |
| `parameter_group_name` | `string` | Name of the created parameter group. |

### Usage

```hcl
module "rds_platform" {
  source = "git::https://github.com/luminary/infra.git//modules/luminary-rds?ref=modules/luminary-rds/v1.4.2"

  identifier   = "luminary-platform-production"
  environment  = "production"
  instance_class = "db.r6g.2xlarge"
  multi_az     = true

  subnet_ids = module.vpc.data_subnet_ids
  vpc_id     = module.vpc.vpc_id

  allowed_security_group_ids = [
    module.eks.node_security_group_id,
  ]

  allocated_storage_gb     = 200
  max_allocated_storage_gb = 2000
  backup_retention_days    = 14
  deletion_protection      = true

  extra_parameters = {
    "log_min_duration_statement" = "1000"   # log queries > 1 s
    "shared_preload_libraries"   = "pg_stat_statements"
    "max_connections"            = "500"
  }
}
```

### Known Caveats

- `deletion_protection = true` **default:** You must set `deletion_protection = false` before Terraform can destroy the instance. This is intentional. A CI gate prevents `deletion_protection = false` from being merged to production stacks.
- **Parameter group changes:** Some `extra_parameters` require a database restart to take effect (`pending-reboot` status). The module does not handle this — check the AWS console after applying parameter changes and schedule a maintenance window if needed.
- **Storage autoscaling:** `max_allocated_storage_gb` caps autoscaling but storage cannot be shrunk once expanded. Size `allocated_storage_gb` based on 6-month projections to minimise churn.
- **Credentials management:** The module does NOT manage the master password. Credentials are generated out-of-band, stored in AWS Secrets Manager, and injected via External Secrets Operator. Never pass the master password as a Terraform variable.

## `luminary-kafka`

Provisions an Amazon MSK (Managed Streaming for Apache Kafka) cluster with standard broker configuration, CloudWatch metrics, and VPC connectivity.

**Source:** `modules/luminary-kafka` | **Latest:** `v1.2.0`

### Inputs

| Variable | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| `cluster_name` | `string` | — | Yes | MSK cluster name. |
| `environment` | `string` | — | Yes | Environment name. |
| `kafka_version` | `string` | `"3.6.0"` | No | Kafka version. |
| `broker_instance_type` | `string` | — | Yes | MSK broker instance type (e.g., `kafka.m5.2xlarge`). |
| `broker_count` | `number` | `3` | No | Number of brokers. Must be a multiple of the number of AZs (3). |
| `broker_storage_gb` | `number` | `1000` | No | EBS storage per broker in GB. |
| `subnet_ids` | `list(string)` | — | Yes | One subnet per broker (data subnets, one per AZ). |
| `vpc_id` | `string` | — | Yes | VPC ID for security group. |
| `allowed_security_group_ids` | `list(string)` | `[]` | No | Security group IDs allowed to connect on port 9092/9094. |
| `client_authentication` | `string` | `"iam"` | No | Authentication method: `iam`, `sasl_scram`, or `none`. |

### Outputs

| Output | Type | Description |
| --- | --- | --- |
| `bootstrap_brokers_iam` | `string` | IAM-authenticated bootstrap broker string. |
| `bootstrap_brokers_tls` | `string` | TLS bootstrap broker string (for non-IAM clients). |
| `zookeeper_connect_string` | `string` | ZooKeeper connection string (legacy; prefer KRaft-based topics). |
| `cluster_arn` | `string` | MSK cluster ARN (for IAM policies). |

### Usage

```hcl
module "kafka" {
  source = "git::https://github.com/luminary/infra.git//modules/luminary-kafka?ref=modules/luminary-kafka/v1.2.0"

  cluster_name           = "luminary-events-production"
  environment            = "production"
  kafka_version          = "3.6.0"
  broker_instance_type   = "kafka.m5.2xlarge"
  broker_count           = 3
  broker_storage_gb      = 2000

  subnet_ids = module.vpc.data_subnet_ids
  vpc_id     = module.vpc.vpc_id

  allowed_security_group_ids = [
    module.eks.node_security_group_id,
  ]

  client_authentication = "iam"
}
```

### Known Caveats

- **Broker storage cannot be reduced.** MSK broker storage can only be scaled up. Provision based on 12-month growth projections. Current production growth rate: ~80 GB/broker/month.
- `broker_count` **must be a multiple of AZ count (3).** The module enforces this via a `precondition` check, but Terraform will error at plan time if violated.
- **KRaft mode:** Kafka 3.6+ supports KRaft (ZooKeeper-free) mode. The MSK managed service is working toward KRaft GA; the `zookeeper_connect_string` output will be removed in a future module version. Do not build new consumer logic against ZooKeeper; use the `bootstrap_brokers_iam` output exclusively.
- **Topic management:** The module does NOT manage Kafka topics. Topics are provisioned by the Topic Provisioner service (see [Data Flow — Per-Workspace Topic Provisioning](Data-Flow-—-Event-Ingestion-to-Storage.md#per-workspace-topic-provisioning)).

## `luminary-redis`

Provisions an ElastiCache Redis replication group (cluster mode disabled, one primary + one read replica per AZ) for the Luminary query cache and enrichment store.

**Source:** `modules/luminary-redis` | **Latest:** `v1.1.3`

### Inputs

| Variable | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| `cluster_id` | `string` | — | Yes | ElastiCache replication group ID. |
| `environment` | `string` | — | Yes | Environment name. |
| `node_type` | `string` | — | Yes | ElastiCache node type (e.g., `cache.r7g.xlarge`). |
| `num_cache_clusters` | `number` | `2` | No | Number of nodes (1 primary + N-1 replicas). |
| `engine_version` | `string` | `"7.1"` | No | Redis engine version. |
| `subnet_ids` | `list(string)` | — | Yes | Subnet IDs for the cache subnet group. |
| `vpc_id` | `string` | — | Yes | VPC ID for security group. |
| `allowed_security_group_ids` | `list(string)` | `[]` | No | Security group IDs allowed to connect on port 6379. |
| `at_rest_encryption` | `bool` | `true` | No | Enable AES-256 at-rest encryption. |
| `transit_encryption` | `bool` | `true` | No | Enable TLS in-transit. |
| `automatic_failover` | `bool` | `true` | No | Enable automatic primary failover. |
| `snapshot_retention_days` | `number` | `3` | No | Number of daily snapshots to retain. |

### Outputs

| Output | Type | Description |
| --- | --- | --- |
| `primary_endpoint` | `string` | Primary endpoint for read/write. |
| `reader_endpoint` | `string` | Reader endpoint for read-only replicas. |
| `port` | `number` | Redis port (6379). |
| `security_group_id` | `string` | Security group ID for the cluster. |

### Usage

```hcl
module "redis_cache" {
  source = "git::https://github.com/luminary/infra.git//modules/luminary-redis?ref=modules/luminary-redis/v1.1.3"

  cluster_id   = "luminary-cache-production"
  environment  = "production"
  node_type    = "cache.r7g.xlarge"
  num_cache_clusters = 3   # 1 primary + 2 replicas, one per AZ

  engine_version = "7.1"

  subnet_ids = module.vpc.data_subnet_ids
  vpc_id     = module.vpc.vpc_id

  allowed_security_group_ids = [
    module.eks.node_security_group_id,
  ]

  at_rest_encryption    = true
  transit_encryption    = true
  automatic_failover    = true
  snapshot_retention_days = 3
}
```

### Known Caveats

- **Cluster mode disabled:** The module provisions a single-shard replication group (cluster mode disabled). This limits total keyspace to the memory of a single node. The current `cache.r7g.xlarge` (26 GB RAM) is sufficient for the query cache workload; if we need to scale past ~20 GB of active cache data, the module will need to be extended to support cluster mode.
- `transit_encryption = true` **requires TLS-aware clients.** All Luminary services use the `go-redis` client with TLS configured. If you add a new consumer, ensure you configure `TLSConfig` on the Redis client.
- **Failover time:** In the event of a primary node failure, automatic failover takes 30–60 seconds. During this window, all writes fail. Services using Redis for the query cache handle this by falling back to direct ClickHouse queries (degraded but functional). The Billing Service's usage counter is buffered in Kafka and reconciled on reconnect.
