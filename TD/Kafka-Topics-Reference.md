# Kafka Topics Reference

This document is the complete reference for all Kafka topics managed by the Data Engineering team on Luminary's MSK cluster (`luminary-msk-prod`). It covers topic configuration, message schemas (Avro), consumer groups, and operational notes.

**Cluster**: `luminary-msk-prod` (MSK, 3 brokers, `kafka.3xlarge`) **Kafka version**: 3.5.1 **Schema Registry**: Confluent Schema Registry at `https://schema-registry.luminary.internal` **Default replication factor**: 3 (all brokers)

## luminary.events.raw

**Purpose**: Entry point for all inbound event data. The Ingest API writes every accepted event here immediately after basic HTTP validation (auth check, payload size check). No schema or business validation has occurred at this stage.

### Configuration

```properties
# Topic config
num.partitions=64
replication.factor=3
retention.ms=172800000          # 48 hours
retention.bytes=-1              # unlimited (time-based only)
min.insync.replicas=2
compression.type=lz4
message.max.bytes=2097152       # 2 MB max message size
```

### Partition Assignment

Events are partitioned by `workspace_id` (hashed). This ensures all events from a single workspace land on the same partition and are therefore processed in order by downstream consumers. 64 partitions currently provides adequate parallelism; the target is to keep each partition under 300 msg/sec sustained.

### Consumer Groups

| Group ID | Service | Lag SLO |
| --- | --- | --- |
| `validation-service-prod` | Validation Service — reads raw, validates schema, writes to `.validated` or `.dlq` | < 5,000 messages |

## luminary.events.dlq

**Purpose**: Dead-letter queue for events that failed validation or enrichment. Events land here with an attached `failure_reason` and `failed_stage` field. The DLQ has a 14-day retention so on-call engineers have time to investigate and replay if warranted.

### Configuration

```properties
num.partitions=8
replication.factor=3
retention.ms=1209600000         # 14 days
min.insync.replicas=2
compression.type=lz4
```

### DLQ Alerting

A Datadog monitor fires if the DLQ message rate exceeds 100 msg/min sustained for 3 minutes. Common causes:

- **Schema violations**: Customer SDK sending events with malformed property types. Usually self-resolving if caused by a bad deploy on customer side.
- **Workspace not found**: Events arriving for a deleted or suspended workspace. These are expected and do not need action unless the rate is unusual.
- **ClickHouse write failure**: Connection pool exhaustion or ClickHouse node unavailability. Check `clickhouse-writer` pod logs immediately.

### Retry Backoff Schedule

| Attempt | Delay |
| --- | --- |
| 1 (initial) | immediate |
| 2 | 30 seconds |
| 3 | 5 minutes |
| 4 | 30 minutes |
| 5 | 2 hours |
| After attempt 5 | Marked `failed`, customer notified via in-app alert |

## Operational Procedures

### Checking Consumer Lag

```shell
# All groups on the events pipeline
kafka-consumer-groups.sh \
  --bootstrap-server kafka.luminary.internal:9092 \
  --command-config /etc/kafka/client.properties \
  --describe \
  --group enrichment-service-prod

# Quick lag summary across all monitored groups
# (wrapper script in data-platform/scripts/kafka-lag-summary.sh)
./scripts/kafka-lag-summary.sh --threshold 50000
```

### Replaying DLQ Messages

```shell
# Replay a range of DLQ messages back to luminary.events.raw
# Requires on-call Kafka access role
./scripts/dlq-replay.sh \
  --source luminary.events.dlq \
  --target luminary.events.raw \
  --workspace-id 018e2f1a-b3c4-7d8e-9f0a-1b2c3d4e5f6a \
  --from-offset 48291 \
  --to-offset 48350 \
  --dry-run    # remove to actually replay
```

### Adding a New Topic

Follow the naming convention `luminary.<domain>.<entity>` (e.g. `luminary.reports.export_jobs`). All new topics require:

1. A PR to `data-platform/kafka/topics/` with the topic config YAML.
2. An Avro schema registered in the Schema Registry.
3. An entry in this reference document.
4. Approval from the owning team lead.

See [Working with Kafka](https://placeholder.invalid/page/developer-guide%2Fworking-with-kafka.md) for developer guidance.
